# Worker Retention And R2 Purge Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Phase 5C Worker scheduled maintenance: retain event/run tables according to the design document and complete the delayed remote delete flow by purging due R2 objects and marking episodes as `purged`.

**Architecture:** Keep the scheduled task inside the existing Cloudflare Worker. A small exported maintenance function performs three bounded jobs: delete old event rows, delete old completed sync run rows, and purge a limited page of due `delete_pending` episodes. R2 delete happens before the D1 status transition. D1 changes for each purged episode are conditional and write a `tombstone_changes` row only when the status update succeeds.

**Tech Stack:** Cloudflare Workers, TypeScript, D1, R2 binding, Vitest fake D1/R2 tests, existing Wrangler dry-run.

## Acceptance Criteria

- Worker exports a `scheduled` handler that runs maintenance without requiring an HTTP request.
- `wrangler.jsonc` declares a cron trigger and an R2 bucket binding for the media bucket.
- Events older than 30 days are deleted.
- Completed sync runs older than 180 days are deleted.
- Currently running sync runs are not deleted by retention.
- `delete_pending` episodes whose `purge_after <= now` are processed in a bounded page.
- For a due episode with a non-empty `r2_key`, Worker calls `env.MEDIA_BUCKET.delete(r2_key)` before changing D1 status.
- If R2 delete fails, the episode remains `delete_pending` and no purge tombstone is written.
- If D1 conditional update observes a status/r2_key race after R2 delete, no tombstone is written and the scheduled task continues with other candidates.
- If an episode is due and has no `r2_key`, it may be marked `purged` without an R2 delete because there is no remote object to remove.
- Purging a row sets status to `purged`, updates `updated_at`, and clears `purge_after`; it does not make the episode visible.
- Every successful purge writes one `tombstone_changes` row with `status='purged'` and `action='purge'`.
- Scheduled maintenance is best-effort per candidate: one candidate failure must not stop later candidates.
- Existing fetch routes and tests remain compatible.
- `npm run check`, `npm run d1:check`, `npm run wrangler:check`, and `go test ./...` pass.

## Retention And Purge Contracts

Constants:

```ts
const eventRetentionDays = 30;
const syncRunRetentionDays = 180;
const purgeBatchLimit = 50;
```

Use a single `now` timestamp generated by the scheduled invocation, formatted as an ISO string acceptable to D1 `datetime(?)`. Tests should use a fixed `now`.

Retention SQL:

```sql
DELETE FROM events
 WHERE datetime(event_time) < datetime(?, '-30 days')
```

```sql
DELETE FROM sync_runs
 WHERE status <> 'running'
   AND datetime(COALESCE(finished_at, started_at)) < datetime(?, '-180 days')
```

Purge candidate SQL:

```sql
SELECT feed_id, local_episode_id, r2_key
  FROM episodes
 WHERE status = 'delete_pending'
   AND purge_after IS NOT NULL
   AND datetime(purge_after) <= datetime(?)
 ORDER BY datetime(purge_after) ASC, feed_id ASC, local_episode_id ASC
 LIMIT ?
```

Conditional purge update SQL:

```sql
UPDATE episodes
   SET status = 'purged',
       purge_after = NULL,
       updated_at = ?
 WHERE feed_id = ?
   AND local_episode_id = ?
   AND status = 'delete_pending'
   AND (
     (r2_key IS NULL AND ? IS NULL)
     OR r2_key = ?
   )
   AND purge_after IS NOT NULL
   AND datetime(purge_after) <= datetime(?)
```

Tombstone insert SQL:

```sql
INSERT INTO tombstone_changes (feed_id, local_episode_id, status, action, created_at)
SELECT ?, ?, 'purged', 'purge', ?
 WHERE changes() = 1
```

Run update and tombstone insert in a D1 batch and require both changes to be `1` before counting the purge as successful.

## Task 2: Scheduled Maintenance Core

**Files:**

- Modify: `cloudflare/worker/src/index.ts`

- [ ] **Step 2.1: Add maintenance types and constants**

Add small internal interfaces:

```ts
interface PurgeCandidateRow {
  feed_id: string;
  local_episode_id: string;
  r2_key: string | null;
}

interface MaintenanceResult {
  old_events_deleted: number;
  old_sync_runs_deleted: number;
  purge_candidates: number;
  episodes_purged: number;
  purge_errors: number;
}
```

Keep these near other Worker helpers. They are not API response contracts.

- [ ] **Step 2.2: Add `runScheduledMaintenance`**

Create an exported function for tests:

```ts
export async function runScheduledMaintenance(env: Env, now = new Date()): Promise<MaintenanceResult>
```

Implementation order:

1. Format `now` once as `now.toISOString()`.
2. Delete old events.
3. Delete old completed sync runs.
4. Select due purge candidates.
5. Process candidates serially.
6. Return counts.

Use `result.meta.changes ?? 0` for delete/update counts.

- [ ] **Step 2.3: Add `purgeEpisodeCandidate` helper**

Pseudo-flow:

```ts
async function purgeEpisodeCandidate(env: Env, candidate: PurgeCandidateRow, nowISO: string): Promise<boolean> {
  if (candidate.r2_key && !env.MEDIA_BUCKET) return false;
  if (candidate.r2_key) await env.MEDIA_BUCKET.delete(candidate.r2_key);

  const update = env.DB.prepare(purgeEpisodeUpdateSQL()).bind(
    nowISO,
    candidate.feed_id,
    candidate.local_episode_id,
    candidate.r2_key,
    candidate.r2_key,
    nowISO,
  );
  const tombstone = env.DB.prepare(purgeTombstoneInsertSQL()).bind(
    candidate.feed_id,
    candidate.local_episode_id,
    nowISO,
  );
  const [updateResult, tombstoneResult] = await env.DB.batch([update, tombstone]);
  return updateResult?.meta.changes === 1 && tombstoneResult?.meta.changes === 1;
}
```

Do not wrap all candidates in one transaction because R2 delete is external and per-row failure should not block later rows.

- [ ] **Step 2.4: Handle errors per candidate**

If one candidate throws from R2 or D1:

- increment `purge_errors`.
- leave that candidate for a later cron run if D1 status did not change.
- continue processing later candidates.
- use `console.warn` with feed id and local episode id only; do not log tokens or object keys.

The function can throw only for retention-query failures or initial candidate-select failure, because those indicate the whole scheduled job cannot proceed.

- [ ] **Step 2.5: Wire Worker `scheduled` handler**

Add:

```ts
async scheduled(_controller: ScheduledController, env: Env, ctx: ExecutionContext): Promise<void> {
  ctx.waitUntil(runScheduledMaintenance(env).then(
    (result) => console.log("scheduled maintenance completed", result),
    (error) => console.error("scheduled maintenance failed", error),
  ));
}
```

Keep `fetch` behavior unchanged.

Use the module Worker type shape:

```ts
export default {
  async fetch(...) { ... },
  async scheduled(...) { ... },
} satisfies ExportedHandler<Env>;
```

This lets `npm run typecheck` verify the scheduled handler signature against `@cloudflare/workers-types`.

**Files:**

- Modify: `cloudflare/worker/test/fake-d1.ts`

- [ ] **Step 3.1: Extend fake D1 options**

Add optional fields:

```ts
now?: string;
failPurgeUpdateKeys?: Set<string>;
```

The fake already stores episodes, tombstone changes, sync runs, events. Reuse those maps.

- [ ] **Step 3.2: Support retention deletes**

In `FakeStatement.runWithOptions`, detect:

- `DELETE FROM events`
- `DELETE FROM sync_runs`

Implement cutoff comparison from the bound `now` and fixed day counts implied by query text. Delete matching fake rows and return the number of removed rows.

Use the existing `sqliteDateTimeMillis` helper, or an equivalent UTC-normalizing helper, for fake cutoff comparisons. Do not use raw `Date.parse` directly on SQLite strings such as `"2026-07-06 00:00:00"`, because Node treats that shape as local time while D1/SQLite `datetime(...)` uses UTC.

Rules:

- Event retention uses `event_time`.
- Sync run retention uses `finished_at ?? started_at`.
- Sync run retention skips `status === "running"`.

- [ ] **Step 3.3: Support purge candidate select**

In `allRows`, detect the due `delete_pending` candidate query. Return rows where:

- `status === "delete_pending"`
- `purge_after` is non-null
- `sqliteDateTimeMillis(purge_after) <= sqliteDateTimeMillis(nowISO bound)`

Sort by `purge_after`, `feed_id`, `local_episode_id`; apply limit.

- [ ] **Step 3.4: Support purge status update**

In `runEpisodeStatusUpdate`, distinguish purge update from admin delete/restore/hide:

- detect `status = 'purged'`.
- ensure current status is `delete_pending`.
- ensure current `purge_after` is due.
- ensure bound r2 key matches current `r2_key`, including null.
- if key is in `failPurgeUpdateKeys`, return 0 to simulate a race.
- set `status='purged'`, `purge_after=null`, `updated_at=nowISO`.
- return 1.

Existing admin action tests must continue to pass.

- [ ] **Step 3.5: Support purge tombstone insert**

The existing fake tombstone insert path assumes the admin-action shape:

```sql
INSERT INTO tombstone_changes (...) SELECT ?, ?, ?, ?, CURRENT_TIMESTAMP ...
```

The purge path uses SQL literals for status/action and binds only:

```text
[feed_id, local_episode_id, created_at]
```

Extend `runTombstoneInsert` so it recognizes the purge insert query containing both `"'purged'"` and `"'purge'"`. For that shape, use:

- `feedID = params[0]`
- `localEpisodeID = params[1]`
- `status = "purged"`
- `action = "purge"`
- `created_at = params[2]`

Keep the existing admin-action parameter handling unchanged for hide/delete/restore tests.

- [ ] **Step 3.6: Add fake R2 bucket**

In the scheduled maintenance test file, define a minimal fake:

```ts
class FakeR2Bucket {
  deletedKeys: string[] = [];
  failKeys = new Set<string>();
  async delete(key: string): Promise<void> {
    if (this.failKeys.has(key)) throw new Error("delete failed");
    this.deletedKeys.push(key);
  }
}
```

Use type assertion to pass it as `R2Bucket`; no need to implement unrelated R2 methods.
